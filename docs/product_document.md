# AutoVisionTest - AI视觉驱动桌面应用自动化测试框架

## 产品文档 v1.0

---

## 1. 产品概述

### 1.1 产品定位

AutoVisionTest 是一款面向桌面应用的纯黑盒自动化测试框架，采用 AI 视觉模型驱动，通过截图识别和模拟键鼠操作实现自动化测试。核心目标是**配合 AI 编程工作流**，实现"编码 → 测试 → 反馈 → 修复"的闭环。

### 1.2 核心价值

| 价值维度 | 描述 |
|---------|------|
| **零侵入** | 纯黑盒测试，无需修改被测应用代码，无需注入 Agent |
| **AI 驱动** | 基于本地部署的视觉语言模型，理解屏幕内容并自主决策 |
| **闭环集成** | 测试结果结构化输出，直接反馈给 AI 编程 Agent |
| **可扩展** | 支持多种视觉模型和测试策略，适应不同测试场景 |

### 1.3 目标用户

- AI 编程工作流用户（开发者使用 AI 编码工具时，自动触发测试验证）
- 桌面应用 QA 工程师
- 持续集成 / 持续部署流水线

### 2.2 核心模块说明

#### 2.2.1 测试管理器（Scheduler）

测试任务调度中心，负责：
- 接收来自 AI 编程 Agent 的测试请求
- 管理测试用例队列和执行顺序
- 控制测试生命周期（启动、暂停、终止）
- 管理测试环境快照（保存 / 恢复应用状态）

#### 2.2.2 场景解析器（Parser）

将测试描述转化为可执行的测试步骤：
- **确定性用例**：解析结构化测试脚本（YAML），生成固定操作序列 + 断言
- **探索性用例**：解析自然语言测试目标，交由规划 Agent 动态生成步骤

#### 2.2.3 测试执行引擎

核心执行循环，参考 ScreenAgent 的"规划 - 执行 - 反思"模式，并借鉴 UFO 的双 Agent 架构：

| Agent | 职责 | 参考 |
|-------|------|------|
| **规划 Agent** | 分析当前截图，确定下一步操作 | UFO AppAgent |
| **执行 Agent** | 执行具体键鼠操作 | UFO ActAgent |
| **反思 Agent** | 验证操作结果，判断是否达成目标 | ScreenAgent Reflection |

#### 2.2.4 视觉感知层

三级视觉识别策略，确保纯黑盒可用性：

| 级别 | 方式 | 精度 | 依赖 |
|------|------|------|------|
| **L1 像素级** | 截图 + SSIM/SIFT 对比 | 低（仅检测视觉变化） | 无 |
| **L2 OCR 级** | 文字识别 + 元素定位 | 中 | PaddleOCR / Tesseract |
| **L3 模型级** | VLM 视觉语言模型理解 | 高 | ShowUI / Qwen2-VL |

纯黑盒模式下仅使用 L1 + L2 + L3。若被测应用支持 Accessibility API（如 Windows UI Automation），可选择性接入作为增强。

#### 2.2.5 本地 AI 推理服务

| 模型 | 用途 | 推荐模型 | 最低显存 |
|------|------|----------|---------|
| VLM 视觉模型 | 屏幕理解、操作决策 | ShowUI-2B / Qwen2-VL-2B | 8GB |
| 截图对比模型 | 视觉回归检测 | SSIM + SIFT (CPU) | 无 |
| OCR 模型 | 文字识别 | PaddleOCR (CPU) | 无 |

**模型常驻内存优化**：VLM 模型启动后常驻 GPU 内存，避免每次推理重新加载，单步操作延迟目标 < 3 秒。

#### 2.2.6 桌面控制层

参考 Cradle 的通用计算机控制方式：
- **鼠标操作**：移动、点击、双击、右键、拖拽、滚动
- **键盘操作**：单键、组合键、文本输入
- **窗口管理**：窗口聚焦、最大化、最小化、关闭（Win32 API）

## 3. 测试用例体系

### 3.1 用例分类

| 类型 | 描述 | 确定性 | 适用场景 |
|------|------|--------|---------|
| **确定性用例** | 固定操作序列 + 精确断言 | 高 | 回归测试、核心流程验证 |
| **探索性用例** | 自然语言目标，AI 自主探索 | 中 | 新功能探索、冒烟测试 |
| **混合用例** | 关键步骤固定 + 中间步骤 AI 填充 | 中高 | 复杂流程测试 |

### 3.2 用例格式

#### 确定性用例示例（YAML）

```yaml
test_case:
  id: TC-001
  name: "登录功能验证"
  type: deterministic
  
  setup:
    - action: launch_app
      target: "C:\\App\\target.exe"
    - action: wait
      duration: 3000
      
  steps:
    - step: 1
      action: click
      target_description: "用户名输入框"
      target_visual: { region: [0.2, 0.3, 0.6, 0.4] }  # 相对坐标
      
    - step: 2
      action: type
      target_description: "用户名输入框"
      input: "testuser"
      
    - step: 3
      action: click
      target_description: "密码输入框"
      target_visual: { region: [0.2, 0.4, 0.6, 0.5] }
      
    - step: 4
      action: type
      target_description: "密码输入框"
      input: "password123"
      
    - step: 5
      action: click
      target_description: "登录按钮"
      target_visual: { region: [0.4, 0.6, 0.6, 0.7] }
      
  assertions:
    - type: visual_contains_text
      text: "欢迎"
      timeout: 5000
      
  teardown:
    - action: close_app
```

#### 探索性用例示例（YAML）

```yaml
test_case:
  id: TC-002
  name: "文件导出功能冒烟测试"
  type: exploratory
  
  goal: "验证文件导出功能是否可用，从菜单找到导出选项，执行导出，确认文件已生成"
  
  constraints:
    max_steps: 20
    no_dangerous_actions: true  # 禁止删除、格式化等危险操作
    
  success_criteria:
    - "导出文件对话框出现"
    - "文件成功保存到指定目录"
    
  failure_criteria:
    - "出现错误对话框"
    - "应用崩溃或无响应超过10秒"
```

### 3.3 断言类型

| 断言类型 | 描述 | 感知级别 |
|---------|------|---------|
| `visual_contains_text` | 屏幕包含指定文字 | L2 / L3 |
| `visual_matches_template` | 屏幕匹配模板图片 | L1 |
| `visual_region_changed` | 指定区域发生变化 | L1 |
| `visual_region_unchanged` | 指定区域未变化 | L1 |
| `element_exists` | 指定描述的 UI 元素存在 | L3 |
| `ai_judge` | AI 判断目标是否达成 | L3 |
| `no_error_dialog` | 无错误弹窗 | L2 / L3 |

## 4. 工作流程

### 4.1 与 AI 编程的集成流程

```
AI编程Agent 修改代码
       │
       ▼
  编译 / 构建完成
       │
       ▼
  触发测试请求 ──────────────────┐
       │                        │
       ▼                        ▼
  AutoVisionTest          测试管理器
  接收请求                选择关联用例
       │                        │
       ▼                        ▼
  ┌─────────────────────────────────┐
  │        测试执行循环              │
  │                                  │
  │  1. 截图采集                     │
  │  2. VLM 分析当前屏幕             │
  │  3. 规划下一步操作               │
  │  4. 执行键鼠操作                 │
  │  5. 截图验证操作结果             │
  │  6. 反思判断是否继续             │
  │  7. 检查终止条件                 │
  │     ├── 目标达成 → 成功退出      │
  │     ├── 最大步数 → 超时退出      │
  │     ├── 重复动作 → 卡死退出      │
  │     └── 继续执行 → 回到步骤1     │
  └─────────────────────────────────┘
       │
       ▼
  生成测试报告
       │
       ▼
  结构化反馈 ─────────────────► AI编程Agent
  (JSON Protocol)            根据反馈修复代码
```

### 4.2 终止条件设计

| 终止条件 | 描述 | 优先级 |
|---------|------|--------|
| **目标达成** | 反思 Agent 判断测试目标已完成 | 最高 |
| **断言通过** | 所有断言条件满足 | 最高 |
| **最大步数** | 执行步数超过 max_steps 阈值 | 高 |
| **重复动作** | 连续 3 次相同操作无状态变化 | 高 |
| **错误弹窗** | 检测到应用错误弹窗 | 中 |
| **应用崩溃** | 目标窗口不存在 | 最高 |
| **人工干预** | 用户手动终止 | 低 |

## 6. 环境与数据管理

### 6.1 测试环境快照

| 功能 | 描述 |
|------|------|
| **环境初始化** | 测试前自动启动被测应用并导航到初始状态 |
| **数据预设** | 支持通过操作序列准备测试数据 |
| **环境恢复** | 测试结束后关闭应用，清理临时文件 |
| **状态回滚** | 关键步骤前保存快照，失败时可回滚重试 |

### 6.2 硬件需求

| 配置级别 | GPU | RAM | 存储 | 适用场景 |
|---------|-----|-----|------|---------|
| **最低配置** | 8GB 显存 (RTX 3060) | 16GB | 20GB | 2B 模型，基础测试 |
| **推荐配置** | 16GB 显存 (RTX 4080) | 32GB | 50GB | 4B 模型，复杂测试 |
| **高性能配置** | 24GB 显存 (RTX 4090) | 64GB | 100GB | 7B 模型，并发测试 |

## 8. 性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 单步操作延迟 | < 3 秒 | 从截图到操作执行完成 |

| 断言准确率 | > 95% | 判断测试通过 / 失败的准确度 |
